# Repository Instructions

本仓库是“无界知识”的长期知识工程系统和知识网络底座。根定位由 `system/project-ontology.json` 接管；`AGENTS.md` 只作为薄层入口，不承载具体模块说明、特殊流程或局部方法论；这些信息应通过机器友好的路由登记文件接管。

## 入口顺序

1. 先读 `system/route-registry.json`，判断当前任务应该进入哪些文件、模块、schema、workflow 或工具。
2. 再读 `system/information-map.json`，确认模块边界、数据流、存储位置、入口和受影响面。
3. 涉及知识条目检索时，读 `70-indexes/catalog.json`，再回到对应来源记录、笔记、地图、综合产物或工作流。

## 四个基本原则

1. 遇到问题先诊断：从局部现象向上追溯，判断问题属于哪一层；找到更高层原因后，用最少改动、最大复用面收益的方式修复或升级。
2. 系统升级必须协同：修复、需求、能力新增或架构升级都不能只改局部；必须检查相关模块、规则、schema、dataflow、gate、route、workflow、tool map、memory、版本记录等受影响面，并在必要时一起升级。
3. 信息地图与机器友好数据流：系统必须维护 information map；内部数据流优先机器可读、省 token、可引用、可切片、可快速检索。Markdown 只做人类说明，不作为系统行为真相来源。
4. 新系统模块必须对旧内容负责：任何新增或升级出的系统模块，不只接管未来内容，也必须审查其职责范围内已经存在的旧文件、旧图、旧路由、旧规则、旧状态和旧文档，并明确接管、迁移、弃用、归档或登记后续处理。

## 通用约束

- 不要把 API key、cookie、token、账号密码、个人隐私材料写入仓库。
- 不要大段复制论文、网页或图书内容；保存摘要、引用线索、链接和自己的分析。
- 修改知识结构后，按路由登记运行相关校验；默认至少运行 `scripts/validate-system.ps1`、`scripts/build-index.ps1` 和 `scripts/validate-kb.ps1`。
- 如果发现路由、信息地图、schema 或工作流与实际文件不一致，优先修正机器友好登记文件，再更新人类说明文档。
