# 采集与沉淀流程

> 本文是人读流程说明；知识摄入子系统的机器真相由 `system/knowledge-ingestion-system.json` 接管，文档边界由 `system/knowledge-ingestion-docs-map.json` 接管。

## 总流程

1. 捕获原始材料到 `00-inbox/`。
2. 判断材料类型：论文、GitHub、网页、个人草稿、项目经验。
3. 在 `10-sources/` 建立来源记录。
4. 从来源中抽取可复用知识，写入 `20-notes/`。
5. 如果多个来源形成稳定判断，写入 `40-synthesis/`。
6. 如果能指导 Codex 做事，写入 `50-workflows/`。
7. 运行索引和校验脚本。

## 论文采集

每篇论文至少记录：

- 标题、作者、年份、DOI 或 URL。
- 研究问题和方法。
- 主要发现。
- 局限性。
- 对工程或行业认知的影响。
- 可复用概念和主张。

论文不要直接等同于事实。优先拆成“论文来源记录”和“基于该论文支持的原子知识”。

## GitHub 技术主题采集

每个 GitHub 主题或仓库至少记录：

- 仓库地址、owner、license。
- stars、forks、最近 release、最近 commit 的检查日期。
- 解决的问题。
- 技术栈和生态位置。
- 成熟度信号：issue 活跃度、维护频率、文档质量、测试情况。
- 风险信号：未维护、许可证冲突、安全问题、过度营销。

GitHub 指标会变化，所以记录时必须写 `last_checked_at`。

## 个人经验采集

个人经验不需要伪装成外部证据，但要标注清楚：

- 经验场景。
- 当时约束。
- 观察到的结果。
- 可能不适用的边界。
- 是否被外部来源支持。

## 去重规则

- 同一来源只保留一个 source record。
- 同一概念只保留一个主概念卡，别的笔记链接过去。
- 同一结论如果证据更强，更新原文件，不要另起近似标题。
- 过时条目移动到 `90-archive/` 或将 `status` 改为 `superseded`。
